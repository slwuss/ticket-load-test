📦
Booking Service
docker-booking-service-1 · :8080

The public-facing HTTP API. Clients talk only to this service. It never writes to the database directly — it only locks seats in Redis and publishes messages to SQS.

Go + Gin Port 8080 REST API
  • Receives reserve / confirm / cancel requests from clients
  • Acquires a Redis lock (SET NX with 10-min TTL) to prevent double-booking
  • Publishes a BookingMessage to SQS FIFO queue
  • Returns booking_id immediately — does NOT wait for DB write
  • Depends on Redis being healthy before starting
⚡
Redis
docker-redis-1 · :6379

Acts as a distributed lock manager. Prevents two users from booking the same seat at the same time, even under heavy concurrent load.

redis:7-alpine Port 6379 In-memory
  • Key format: seat:lock:{eventID}:{seatID}
  • SET NX — only succeeds if the key does not exist (atomic)
  • TTL of 10 minutes — lock auto-expires if user never confirms
  • Release uses a Lua script to ensure only the lock owner can delete it
  • No persistence (--save "") — locks are ephemeral by design
📨
SQS FIFO Queue (LocalStack)
docker-localstack-1 · :4566

Simulates AWS SQS locally. Acts as a buffer between the Booking Service and Worker. Decouples the API from the database so the API stays fast even if the DB is slow.

LocalStack 3 FIFO Queue Port 4566
  • Two queues: booking-queue.fifo (main) and booking-dlq.fifo (dead letter)
  • FIFO ensures messages for the same event are processed in order (MessageGroupId = eventID)
  • Messages that fail processing reappear after 30-second visibility timeout → auto-retry
  • Created by the init-sqs one-shot container on first start
⚙️
Worker Service
docker-worker-service-1

Background consumer that reads messages from SQS and writes the final state to PostgreSQL. Runs 10 concurrent goroutines to maximize throughput.

Go 10 goroutines Long-poll 20s
  • Polls SQS with long-polling (20s wait) — efficient, not busy-looping
  • Handles 3 actions: reserve → insert row, confirm → update status, cancel → update + release Redis lock
  • On DB error → does NOT delete SQS message → message reappears for retry
  • On success → deletes the SQS message to prevent duplicate processing
  • Depends on Postgres + Redis being healthy before starting
🗄️
PostgreSQL
docker-postgres-1 · :5432

The source of truth for all booking records. Only the Worker Service writes to it. Data persists across container restarts via a named Docker volume.

postgres:16-alpine Port 5432 Persistent volume
  • Database: ticketing · User: ticketing · Password: secret
  • Single table: bookings with unique constraint on (event_id, seat_id)
  • Status values: reserved → confirmed or cancelled
  • Volume pgdata persists data even if the container is removed
🚀
init-sqs
amazon/aws-cli · one-shot

A one-time setup container that creates the two SQS FIFO queues in LocalStack. It runs once on startup and exits — it is not a long-running service.

One-shot Depends on LocalStack
  • Waits for LocalStack to be healthy before running
  • Creates booking-dlq.fifo first, then booking-queue.fifo
  • Both queues use ContentBasedDeduplication=true
  • Other services wait for this container to complete before starting
Request Flows
🔒 Reserve a Seat
  1. Client sends POST /api/v1/events/{event_id}/reserve with seat_id and user_id
  2. Booking Service generates a new UUID as booking_id
  3. Tries SET NX seat:lock:{eventID}:{seatID} in Redis with 10-min TTL
  4. If key already exists → return 409 Conflict — seat is taken
  5. Publishes action: "reserve" message to SQS FIFO queue
  6. Returns 202 Accepted with booking_id immediately
  7. Worker picks up the SQS message and inserts a row in Postgres with status = reserved
✅ Confirm a Booking
  1. Client sends POST /api/v1/bookings/{booking_id}/confirm with event + seat + user
  2. Booking Service publishes action: "confirm" to SQS
  3. Returns 202 Accepted with status: "confirming"
  4. Worker receives message and runs UPDATE bookings SET status = 'confirmed'
  5. Only updates if current status is reserved — prevents double-confirm
  6. Redis lock remains until TTL expires (seat is now permanently booked)
❌ Cancel a Booking
  1. Client sends DELETE /api/v1/bookings/{booking_id}?event_id=&seat_id=
  2. Booking Service immediately releases the Redis lock (DEL seat:lock:...)
  3. Publishes action: "cancel" to SQS
  4. Returns 200 OK — seat is instantly available for others to book
  5. Worker receives message and runs UPDATE bookings SET status = 'cancelled'
  6. Worker also releases Redis lock again (idempotent — safe if already released)
🔁 Retry / Error Handling
  1. Worker fails to process a message (e.g., DB error)
  2. Worker does NOT call DeleteMessage on SQS
  3. SQS visibility timeout expires after 30 seconds
  4. Message becomes visible again and another goroutine picks it up
  5. This continues automatically until the message is processed successfully
  6. Failed messages eventually move to booking-dlq.fifo dead-letter queue

1

HTTP Request

Content-Type: application/json

{

2
⚙️ Booking Service — Internal
Generate booking_id
Service creates a new UUID. This ID links every hop together — Redis, SQS, and Postgres all use it.
Go Code (uuid.New)
// uuid.New().String()
booking_id = "98e15093-3999-499d-a38a-47cc5d5e5763"
3
⚡ Booking Service → Redis
Acquire seat lock (SET NX)
Atomically locks the seat. Only one booking can hold this key at a time.
Redis Command Sent
SET  seat:lock:evt-001:A3
     "98e15093-3999-499d-a38a-47cc5d5e5763"
     NX        ← only set if key does NOT exist
     EX 600   ← auto-expire in 10 minutes

key format → seat:lock:{event_id}:{seat_id}
value      → booking_id (so only owner can unlock)
OK → lock acquired, continue to step 4 nil → seat taken → return 409, stop here
4
📨 Booking Service → SQS
Publish message to queue
Sends a JSON message to the FIFO queue. The Worker will pick this up asynchronously.
SQS SendMessage payload
QueueUrl:               booking-queue.fifo
MessageGroupId:         "evt-001"   ← orders messages per event
MessageDeduplicationId: new UUID    ← prevents SQS duplicates

MessageBody:
{
  "booking_id": "98e15093-3999-499d-a38a-47cc5d5e5763",
  "event_id":   "evt-001",
  "seat_id":    "A3",
  "user_id":    "user-alice",
  "action":     "reserve"
}
OK → continue to step 5 Error → release Redis lock, return 500
5
📦 Booking Service → Client
Return 202 immediately
Does NOT wait for the DB write. Client gets the booking_id right away to use for confirm/cancel.
HTTP Response
HTTP/1.1 202 Accepted
{
  "booking_id": "98e15093-3999-499d-a38a-47cc5d5e5763",
  "status":     "reserved",
  "message":    "seat held for 10 minutes"
}
6
⚙️ SQS → Worker Service
Worker polls and receives the message
10 goroutines are long-polling SQS. One picks up the message. It is hidden from others for 30s (VisibilityTimeout).
SQS ReceiveMessage call
ReceiveMessage(
  QueueUrl:            "...booking-queue.fifo",
  MaxNumberOfMessages: 10,
  WaitTimeSeconds:     20,  ← long-poll, not busy-loop
  VisibilityTimeout:   30   ← hidden from other workers
)

Worker sees action = "reserve" → calls UpsertReserved
7
🗄️ Worker Service → PostgreSQL
Insert row into bookings table
Worker writes the booking to the DB. Uses ON CONFLICT to safely overwrite cancelled rows for the same seat.
SQL Executed
INSERT INTO bookings (id, event_id, seat_id, user_id, status)
VALUES (
  '98e15093-3999-499d-a38a-47cc5d5e5763',
  'evt-001', 'A3', 'user-alice', 'reserved'
)
ON CONFLICT (event_id, seat_id) DO UPDATE SET
  id         = EXCLUDED.id,
  user_id    = EXCLUDED.user_id,
  status     = 'reserved',
  updated_at = NOW();

-- Row in DB after this runs:
-- id         │ 98e15093-3999-499d-a38a-47cc5d5e5763
-- event_id   │ evt-001
-- seat_id    │ A3
-- user_id    │ user-alice
-- status     │ reserved
-- created_at │ 2026-06-19 07:46:45+00
OK → Worker calls DeleteMessage on SQS → done DB Error → skip delete → message retries after 30s

📦 Client → Booking Service
HTTP POST arrives

Content-Type: application/json

{

  "seat_id":  "A3",
  "user_id":  "user-alice"
}
2
📨 Booking Service → SQS
Publish confirm message
No Redis interaction here — the lock is kept alive. Only the DB update is needed.
SQS MessageBody
{
  "booking_id": "98e15093-3999-499d-a38a-47cc5d5e5763",
  "event_id":   "evt-001",
  "seat_id":    "A3",
  "user_id":    "user-alice",
  "action":     "confirm"   ← key field
}
3
📦 Booking Service → Client
Return 202 immediately
HTTP Response
HTTP/1.1 202 Accepted
{
  "booking_id": "98e15093-3999-499d-a38a-47cc5d5e5763",
  "status":     "confirming"   ← not "confirmed" yet
}
4
🗄️ Worker Service → PostgreSQL
Update row to confirmed
The AND status = 'reserved' guard prevents double-confirm. If 0 rows updated → logs error but message is still deleted.
SQL Executed
UPDATE bookings
SET
  status     = 'confirmed',
  updated_at = NOW()
WHERE
  id     = '98e15093-3999-499d-a38a-47cc5d5e5763'
  AND status = 'reserved';  ← guard: must be reserved

-- Before: status = 'reserved'
-- After:  status = 'confirmed'  ← permanent
--         updated_at = NOW()
rows_affected = 1 → success, delete SQS message rows_affected = 0 → already confirmed or not found
1
📦 Client → Booking Service
HTTP DELETE arrives
event_id and seat_id passed as query params (needed to build the Redis key).
HTTP Request
DELETE /api/v1/bookings/98e15093-3999-499d-a38a-47cc5d5e5763
       ?event_id=evt-001&seat_id=A3

No request body needed
2
⚡ Booking Service → Redis
Release lock immediately (Lua script)
Done synchronously before responding. Seat is free the moment this succeeds — another user can reserve it right away.
Redis Lua Script (atomic)
EVAL "
  if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
  end
  return 0
"
  KEYS[1] = seat:lock:evt-001:A3
  ARGV[1] = "98e15093-3999-499d-a38a-47cc5d5e5763"

Gets current value → only deletes if it matches
booking_id. Prevents cancelling someone else's lock.
Returns 1 → lock deleted, seat free Returns 0 → lock already expired (still OK)
3
📨 Booking Service → SQS
Publish cancel message
SQS MessageBody
{
  "booking_id": "98e15093-3999-499d-a38a-47cc5d5e5763",
  "event_id":   "evt-001",
  "seat_id":    "A3",
  "user_id":    "",          ← not needed for cancel
  "action":     "cancel"
}
4
📦 Booking Service → Client
Return 200 — seat already free
HTTP Response
HTTP/1.1 200 OK
{
  "booking_id": "98e15093-3999-499d-a38a-47cc5d5e5763",
  "status":     "cancelled"
}

Redis lock already deleted in step 2.
Anyone can reserve A3 right now.
5
🗄️ Worker Service → PostgreSQL + Redis
Update DB row + release lock (idempotent)
Worker updates the DB row and also runs the Lua DEL again — safe even if the lock is already gone.
SQL Executed
UPDATE bookings
SET
  status     = 'cancelled',
  updated_at = NOW()
WHERE id = '98e15093-3999-499d-a38a-47cc5d5e5763';

-- Then runs the same Lua DEL on Redis again
-- (returns 0 if already gone — no error)

-- Before: status = 'reserved'
-- After:  status = 'cancelled'
OK → delete SQS message → flow complete

internal/lock/redis.go
— distributed seat locking via Redis
AcquireSeat — lock a seat
// Key format
key = "seat:lock:evt-001:A3"

// SET NX = set only if Not eXists
// Atomic — no race condition possible
ok, err := rdb.SetNX(ctx, key, bookingID, 10*time.Minute)

// ok = true  → key was new → lock acquired
// ok = false → key existed → ErrAlreadyLocked

// TTL = 10 minutes
// Lock auto-deletes if user never confirms
// Seat becomes free again automatically
ReleaseSeat — unlock a seat safely
// Cannot use simple DEL — what if lock expired
// and someone else re-locked the seat?
// We'd delete their lock by mistake!

// Lua script runs atomically on Redis
// GET + DEL in a single operation
script = `
  if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
  end
  return 0
`

// KEYS[1] = "seat:lock:evt-001:A3"
// ARGV[1] = booking_id (the owner)
// Only deletes if current value = our booking_id
ExtendSeat — reset the TTL (keep-alive)
// Used to extend the 10-min window
// if the user is still on the checkout page

script = `
  if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
  end
  return 0
`

// ARGV[2] = 600 (seconds = 10 min)
// Only resets TTL if we are still the owner
// Note: not called anywhere yet — reserved

Error Handling Summary
Scenario Where it happens What the service does HTTP Response
Missing seat_id or user_id in body handler — ShouldBindJSON Gin auto-rejects with validation error message 400
Seat already locked by another user lock — AcquireSeat returns ErrAlreadyLocked Returns "seat already taken", does not publish to SQS 409
Redis is down lock — SetNX returns error Returns "failed to reserve seat" 500
SQS publish fails after Redis lock acquired handler — Publish returns error Releases the Redis lock first, then returns error (seat is freed) 500
Cancel without event_id or seat_id handler — c.Query() check Returns "event_id and seat_id required" 400
Redis is down during cancel lock — ReleaseSeat returns error Error is ignored (_ =), continues to publish cancel to SQS anyway 200
Service receives SIGTERM (docker stop) main — signal handler Waits up to 15s for in-flight requests to finish before exiting — graceful
Why is the API asynchronous? The Booking Service returns immediately after locking Redis and publishing to SQS. It does not wait for the Worker to write to Postgres. This keeps API latency low under load — the database write happens in the background. The tradeoff is eventual consistency: right after reserving, the booking may not yet appear in Postgres, but it will within seconds once the Worker processes the SQS message.
