postgres

Durable booking store

Image postgres:16-alpine — mirrors production RDS engine version
Port 5432 — exposed so you can connect with psql or a GUI locally
Role Permanent record of every booking. Worker-service inserts rows here after SQS processing.
Volume pgdata — data persists across docker compose down / up restarts
Health check pg_isready -U ticketing every 5 s — other services wait for this before starting
Prod equivalent RDS PostgreSQL Multi-AZ with read replicas + connection pooling
redis

Distributed seat lock

Image redis:7-alpine — no disk persistence (--save "") so it stays fast
Port 6379
Why Redis? A seat can only be held by one user at a time. Redis SET NX (set-if-not-exists) is atomic — no two booking-service pods can claim the same seat simultaneously.
Lock key seat:lock:evt-001:A42 with a 10-minute TTL — lock auto-releases if payment never arrives
No persistence Locks are intentionally ephemeral. If Redis restarts, locks expire — the SQS dedup ID and Postgres ON CONFLICT prevent double-booking anyway.
Prod equivalent ElastiCache Redis Cluster mode, 3 shards
localstack

Fake AWS SQS — no cloud cost

Image localstack/localstack:3
Port 4566 — every AWS SDK call is redirected here via AWS_ENDPOINT_URL
What it emulates SQS only (SERVICES: sqs). Behaves identically to AWS SQS FIFO — same API, same JSON shapes.
Why FIFO? Message order is guaranteed per MessageGroupId. Each seat gets its own group, so reserve always arrives before confirm.
Health check awslocal sqs list-queues — init-sqs waits for this to pass before creating queues
Prod equivalent Real AWS SQS FIFO queue + DLQ in ap-southeast-2
init-sqs

One-shot queue provisioner

Image amazon/aws-cli — just the CLI, nothing else running long-term
What it does Runs two aws sqs create-queue calls then exits (code 0). Creates booking-dlq.fifo first, then booking-queue.fifo.
booking-dlq.fifo Dead-letter queue. Messages land here after 3 failed processing attempts — nothing is ever silently lost.
booking-queue.fifo Main work queue. booking-service publishes here; worker-service polls and processes messages.
Dependency Waits for localstack: service_healthy before running — guarantees queues exist before any service sends a message
Prod equivalent Terraform aws_sqs_queue resource in terraform/sqs.tf
booking-service

HTTP API — the user-facing layer

Built from ../booking-service — Go binary compiled in Docker, runs on distroless image
Port 8080 — call it with curl http://localhost:8080/api/v1/events/:id/reserve
Step 1 — Lock Calls Redis SET NX seat:lock:… with 10 min TTL. Returns 409 immediately if seat is taken.
Step 2 — Queue Publishes {action:"reserve", seat_id, user_id, booking_id} to booking-queue.fifo and returns 202 booking_id.
Confirm A second call publishes {action:"confirm"} to the same queue — worker picks it up and finalises the DB row.
Dependencies Starts only after redis: healthy + init-sqs: completed
worker-service

SQS consumer — the database writer

Built from ../worker-service — Go binary, long-running process (no HTTP port exposed)
What it does Polls booking-queue.fifo in a loop. For each message it dispatches on action.
reserve action INSERT INTO bookings … ON CONFLICT DO NOTHING — idempotent, safe to retry
confirm action UPDATE bookings SET status='confirmed' — finalises the row created by reserve
Decoupling booking-service never touches Postgres. SQS absorbs bursts — 20 000 VU load test only stresses the queue, not the DB directly.
Dependencies postgres: healthy + redis: healthy + init-sqs: completed

summary

Each service in one line

🗄️postgres

Permanent store of every booking record. Idempotent writes via ON CONFLICT.

⚡redis

Atomic distributed lock that prevents two users reserving the same seat.

📬localstack

Local SQS emulator — same API as AWS, zero cloud cost during dev.

🔧init-sqs

Runs once at startup to create the two FIFO queues, then exits cleanly.

🌐booking-service

HTTP API. Locks seats in Redis, publishes events to SQS, never writes to Postgres.

⚙️worker-service

Background consumer. Drains SQS and writes confirmed bookings to Postgres.
